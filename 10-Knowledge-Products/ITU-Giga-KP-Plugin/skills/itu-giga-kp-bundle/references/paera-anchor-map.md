Use this map when citing PAERA in any subtopic. The map is organised by methodological concept first (the way you typically think about it) and then by section number (the way PAERA is structured). Both views are available — pick whichever is faster for the task at hand.

## Citation patterns for slides and scripts

When citing PAERA in the script body, use natural language followed by the section reference in parentheses or at end of sentence:

- *"The Once-Only principle (PAERA §5.2 Principle #5) says the state never asks a citizen for the same information twice."*
- *"PAERA §3.3 sets out ten principles for Digital Infrastructure — rule of law, intrinsic security, openness, and others."*
- *"The lifecycle — Discover, Assess, Adapt, Plan, Execute and Govern — is described in PAERA §5.4 (Organisational Assessment & Roadmap)."*

For the "Sources" slide that closes each subtopic, list PAERA sections in bullet form:

```
Sources slide. Body: bullet list
- PAERA v1.0 §X.Y (concept name)
- PAERA v1.0 §X.Z (concept name)
Footer: 'Find the link in the description.'
```

The footer instructs viewers to use the YouTube video description for the PAERA URL — do not read URLs aloud per ITU's Knowledge Products Guide.

## What PAERA does NOT cover (do not cite)

Some methodological concepts are not in PAERA and should be cited elsewhere:

- **Four-layer interoperability model (Technical / Semantic / Organisational / Legal)** — Cite **EU European Interoperability Framework (EIF)** and **NIIS X-Road documentation**, not PAERA. PAERA §3.4.3 references interoperability but the four-layer model is from EIF.
- **DPI maturity scoring (Level 0 → Level 5)** — Cite **UNDP DPI Maturity Model**, not PAERA. PAERA §3 frames DPI but the five-level scale is UNDP's.
- **Investment benchmarks (ID4D, fast-payment systems)** — Cite **World Bank ID4D** and **ITU DPI Safeguards**, not PAERA.
- **Building Block specifications and reference implementations** — Cite **GovStack BB specs** (govstack.global), not PAERA. PAERA Annex A1.3 references BBs at the architectural-pattern level; the actual specifications are GovStack assets.
- **Joget DX 8.x patterns** — Internal team experience plus published Joget documentation. Not PAERA.

---

## Anti-pattern: do not invent PAERA section numbers

If a concept does not have a clear PAERA anchor, find a different public reference rather than fabricate a section number. Reviewers will check. A wrong PAERA citation damages the entire bundle's credibility.

When in doubt, the safer move is to cite the concept descriptively without a section reference, or to switch to a verifiable alternative anchor (GovStack, EIF, NIIS, UNDP). PAERA is one anchor among several — it does not have to carry every claim.
